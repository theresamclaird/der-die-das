# Phase 1 — Auth + sync (AWS Amplify Gen 2)

This is the deploy guide for the cross-device sync backend. The app stays fully
usable **without** any of this — study is local-first and works logged-out. Sync
only switches on once a backend is deployed *and* you sign in.

## What's in the repo

- `amplify/` — the Gen 2 backend definition (infrastructure-as-code):
  - `auth/resource.ts` — Cognito, email + password login.
  - `data/resource.ts` — DynamoDB-backed models: `CardState`, `ReviewLog`,
    `UserSettings`, each owner-scoped.
  - `backend.ts` — wires them together.
- `src/lib/amplifyConfig.js` — configures Amplify **only if** `amplify_outputs.json`
  exists; otherwise the app runs local-only (no build error, no AWS needed).
- `src/lib/sync.js` — backend-agnostic `SyncEngine` (pull = last-write-wins on the
  FSRS `last_review`; push = first-run full push, then dirty-only; append-only log
  by cursor).
- `src/lib/amplifyAdapter.js` — the Amplify implementation of the sync interface.
  All `aws-amplify` imports are isolated here.
- `src/Auth.jsx` — minimal email sign-in/up/confirm modal.

## One-time prerequisites

1. An AWS account.
2. Local AWS credentials with permission to deploy (Amplify uses your default
   profile). See Amplify's "Configure AWS for local development" docs.
3. Install deps: `cd web && npm install`.

## Deploy a personal cloud sandbox (development)

```bash
cd web
npx ampx sandbox
```

This provisions Cognito + DynamoDB in your account and writes
`web/amplify_outputs.json` (git-ignored). Leave it running; it watches `amplify/`
and redeploys on change. In another terminal:

```bash
npm run dev
```

The app detects `amplify_outputs.json` on load, so a **"Sign in to sync across
devices"** link appears in the footer. Create an account, confirm via the emailed
code, and your local progress is pushed on first sync.

## Production deploy

Use a Git-based pipeline (Amplify Hosting) or `npx ampx pipeline-deploy`. The
`deploy` script in `package.json` is a starting point (`--app-id` via the
`AMPLIFY_APP_ID` env var). `amplify_outputs.json` is generated by the pipeline at
build time — do not commit it.

## How sync behaves (acceptance criteria, DESIGN §5–§6)

- **Local-first:** every review writes to IndexedDB synchronously; `sync()` runs
  opportunistically (on sign-in, app open while authed, and the `online` event).
  Network is never on the critical path of a review.
- **First sign-in:** all existing local cards are pushed (so offline-only history
  isn't lost), then only changed (`dirty`) cards thereafter.
- **Cross-device:** device A's reviews appear on device B after a sync;
  conflicts resolve last-write-wins on `last_review`.
- **Append-only log:** review rows are immutable and pushed by cursor, so they
  never conflict or duplicate. Cram reps carry `cram: true`.

## Notes / follow-ups

- `UserSettings` model exists but the client currently syncs card state + the
  review log; wiring settings (active levels, toggles, theme) through is a small
  next step.
- Amplify Gen 2 APIs evolve — if `npx ampx sandbox` reports a mismatch, check the
  current docs at https://docs.amplify.aws and bump the `@aws-amplify/*` versions
  in `package.json`.
